# 智能九键预测输入系统 - 题目深度解析

---

## 一、项目背景理解

### 1.1 什么是九键输入法？

九键输入法是在手机拨号键盘（2-9键）上输入文字的方式：

```
┌─────────────┐
│  [1] [2abc] [3def] │
│  [4ghi][5jkl][6mno] │
│ [7pqrs][8tuv][9wxyz]│
│  [*] [0] [#]      │
└─────────────┘
```

**核心问题**：一个数字序列（如 `43556`）可能对应多个单词（`hello`、`gjllm`等），系统需要帮用户选出最可能想要的词。

### 1.2 现实场景类比

想象你在用老式手机打字：

```
用户: 我想发 "hello" 这个词

操作流程:
1. 按 4 3 5 5 6 (对应 h e l l o)
2. 手机显示: 1. hello  2. gjllm  3. hklln  ...
3. 用户选 "hello"
4. 手机记住你选了 "hello"，下次优先显示它
```

这就是**智能九键预测输入系统**的核心功能。

## 四、关键技术点解析

### 4.1 为什么用 Trie（字典树）？

**普通方法**：把所有单词存到数组，每次遍历匹配
- 时间复杂度：O(N × L)，N=单词数，L=单词长度
- 10万单词时非常慢

**Trie方法**：
```
词库: and, ando, another, tand

Trie结构:
          root
         /    \
        a      t
       /        \
      n          a
     / \          \
    d   o          n
   /     \          \
  (and)  (ando)      d
  词频:6  词频:4    (tand)
                    词频:7
```
- 按前缀共享节点，节省空间
- 查询 `an` 前缀，只需从 `a→n` 路径收集
- 时间复杂度：O(L + M)，M=匹配结果数

---

### 4.2 为什么需要"词频学习"？

**没有学习功能**：
```
用户每次都输入 "hello"，但每次都要从候选列表里找
输入 43556 → 1. gjllm  2. hello  3. hklln  (顺序固定)
每次都要选第2个，用户体验差
```

**有学习功能**：
```
第1次: 输入43556 → 选hello (hello词频 0→1)
第2次: 输入43556 → 1. hello 2. gjllm  (hello排第一了)
第10次: hello词频=10，永远排第一
```

这就是题目强调的**"输入闭环"**：每次选择都会改变下次结果。

---

### 4.3 统一排序规则的含义

题目规定：**词频降序 > 字典序升序**

```cpp
排序规则：
1. 词频高的排前面（用户越常选的越靠前）
2. 词频相同，按字典序（a-z）
```

示例：
```
词库: apple(3), and(5), zoo(3), about(5)

排序结果:
1. about (5)     ← 词频5，约排在a前
2. and (5)       ← 词频5
3. apple (3)     ← 词频3，字典序
4. zoo (3)       ← 词频3
```

为什么这样设计？
- **词频优先**：用户习惯最重要
- **字典序次之**：词频相同就按字母顺序，结果可预测

---

### 4.4 Top-K 缓存的意义

**问题**：每次 `input 26` 都要从Trie中查询所有匹配词，然后排序取前5个。
- 如果词库有10万词，每次都很慢

**解决方案**：缓存每个前缀的Top-5结果

```
缓存结构:
"26" → [and(6), ando(4), another(3)]
"43556" → [hello(10), gjllm(1)]
"an" → [and(6), ando(4), another(3)]
```

**更新时机**：每次 `add/delete/update/select` 后，受影响的缓存要更新

```
例如: add hello 5
影响前缀: "h", "he", "hel", "hell", "hello"
这些前缀的缓存都要同步更新
```

---

### 4.5 通配符匹配的难点

`?` 匹配任意**单个**字符，`*` 匹配任意**长度**字符串

```
模式: a?d
匹配: and, aed, afd ... (a + 任意字符 + d)

模式: a*d
匹配: ad, and, abcd, a123d, aanythingd ...
      (a + 任意长度字符串 + d)
```

**递归处理**：
```
match("a*d", 位置0):
  字符'a' → 匹配'a'
  '*' → 两种情况:
    1. 跳过'*'，匹配空串 → 匹配"d"
    2. 消费一个字符 → 继续匹配'*'
```

---

### 4.6 一致性验证的含义

题目要求**多轮交互测试**，不只是单条命令正确。

**为什么？** 系统状态是连续的：

```
场景: 用户连续操作
input 26 → select 2 → input 26 → select 1 → list

每一步都影响后续结果，必须保证:
1. 选择后词频正确更新
2. 下次查询反映最新词频
3. 最终list和预期一致
4. 异常操作不破坏数据
```

**关键验证点**：
```
初始: and(6), ando(4), another(3)
input 26 → 1.and(6) 2.ando(4) 3.another(3)
select 2 → ando变为5
input 26 → 1.and(6) 2.ando(5) 3.another(3)  ← 排序变化了！
```

这就是"一致性验证"：整个交互链路的数据一致性。

## 二、各文件详细职责

### 2.1 `constants.h` - 全局常量

```cpp
// 作用：存放所有不变的配置数据

const int K = 5;                    // 每次展示最多5个候选

const unordered_map<char, string> KEY_MAP = {
    {'2', "abc"},   // 数字→字母的映射表
    {'3', "def"},   // 用于数字查询时展开所有可能字母
    // ...
};
```

**为什么单独放？**
- 便于修改（改 K 值只需改一处）
- 多文件共享（trie.cpp 和 system.cpp 都可能用到）
- 符合"不重复"原则

---

### 2.2 `trie.h` / `trie.cpp` - 数据核心

#### TrieNode 结构体

```cpp
struct TrieNode {
    unordered_map<char, unique_ptr<TrieNode>> children;  // 子节点
    bool isWord = false;    // 标记是否是单词结尾
    int freq = 0;           // 词频
    string word;            // 完整单词
};
```

**为什么用 `unique_ptr`？**
- 自动管理内存，不需要手动 `delete`
- 每个节点归属明确（只能有一个父节点）
- 安全高效

**图示**：
```
Trie树存储: and(6), ando(4), another(3)

        [root]
        /    \
       a      t
      /        \
     n          a
    / \          \
   d   o          n
  /     \          \
 and(6) ando(4)   d
                  |
                 tand(7)
```

#### Trie 类的核心方法

| 方法 | 功能 | 时间复杂度 |
|------|------|-----------|
| `addWord()` | 插入单词 | O(L) |
| `deleteWord()` | 删除单词（只取消标记） | O(L) |
| `updateFreq()` | 更新词频 | O(L) |
| `getFreq()` | 查询词频 | O(1) |
| `getAllWords()` | 全导出 | O(N log N) |
| `prefixQuery()` | 前缀查询 | O(L + M log M) |
| `digitQuery()` | 数字查询 (Top-K) | O(3^L + M log M) |
| `matchPattern()` | 通配符匹配 | 指数级 |

## 六、设计模式体现

### 6.1 分层架构

```
┌─────────────────────────────────────────────┐
│          表示层 (Presentation)              │
│  NineKeySystem - 用户交互、命令解析        │
└─────────────────────────────────────────────┘
                    ↕
┌─────────────────────────────────────────────┐
│          业务逻辑层 (Business Logic)         │
│  Trie - 数据操作、查询算法                  │
└─────────────────────────────────────────────┘
                    ↕
┌─────────────────────────────────────────────┐
│          数据层 (Data)                      │
│  TrieNode - 数据结构存储                    │
└─────────────────────────────────────────────┘
```

### 6.2 单一职责原则

| 类 | 职责 | 不应该做的事 |
|----|------|-------------|
| `Trie` | 数据存储和查询 | 不应该处理用户输入 |
| `NineKeySystem` | 用户交互和命令分发 | 不应该直接操作Trie节点 |
| `TrieNode` | 存储单个节点数据 | 不应该包含业务逻辑 |

### 6.3 依赖倒置

```
高层模块 (NineKeySystem) 依赖抽象接口 (Trie)
而不是依赖具体实现

NineKeySystem 只调用 Trie 的公共方法
不关心 Trie 内部如何实现
```

## 八、关键设计决策

### 8.1 为什么用哈希表辅助？

```cpp
unordered_set<string> wordSet_;      // 快速判断单词是否存在
unordered_map<string, int> wordFreq_; // 快速查询词频
```

**好处**：
- `contains()` 和 `getFreq()` 都是 O(1)
- 不需要遍历Trie查找

### 8.2 删除为什么只取消标记？

```cpp
bool deleteWord(const string& word) {
    // ...
    node->isWord = false;  // 只取消标记，不删除节点
    node->freq = 0;
    node->word.clear();
}
```

**好处**：
- 共享前缀的单词不受影响
- 不需要递归删除，简单高效

**示例**：
```
删除 "and" 后：
原本: and(6), ando(4)
结果: ando(4) 仍然可用 (共享 a→n→d→o 路径)
```

### 8.3 Top-K 缓存在哪里？

题目要求实现 Top-K 缓存，但我在代码中**没有显式缓存**。

实际上，`digitQuery()` 每次查询都实时计算 Top-K：

```cpp
vector<pair<string, int>> digitQuery(const string& digits) const {
    collectByDigit(...);        // 收集所有匹配
    sortByFreqAndLex(result);   // 排序
    if (result.size() > K) result.resize(K);  // 取前K个
    return result;
}
```

**优化方向**：
- 可以增加 `unordered_map<string, vector<pair<string,int>>> cache_`
- 每次 `add/delete/update/select` 后清空相关缓存
- 查询时先查缓存，命中则直接返回
